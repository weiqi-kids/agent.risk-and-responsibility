# L1 | 關鍵事實

**rule_type**: draft

**effective_date**: 公開意見徵詢至 2026-01-30

**原文摘要**:
NIST has released the initial public draft of Special Publication (SP) 800-218r1 (Revision 1), Secure Software Development Framework (SSDF) Version 1.2: Recommendations for Mitigating the Risk of Software Vulnerabilities, per Executive Order 14306

---

# L2 | 權責轉移

**shift_type**: expanded

**enforcement_signal**: recommended

**主體 (actors)**:
- 軟體生產者（減少已發布軟體的漏洞數量，降低未修補漏洞被利用的影響）
- 軟體採購方（在採購流程中與供應商建立共同溝通語彙）
- SDLC實施者（將高層次安全開發實踐整合至各類軟體開發生命週期模型）

**轉移內容**:
SSDF v1.2 相較前版新增及改進的實踐、任務與範例，擴大軟體生產者在安全開發生命週期中的責任範圍。框架要求軟體生產者不僅需減少漏洞數量，更需從根本原因著手防止未來重演，並建立與採購方的共同詞彙以促進供應鏈安全溝通。

**依據條文/段落**:
依據 Executive Order 14306 發布；框架適用於各種 SDLC 模型（waterfall, agile, DevSecOps 等）

---

# L3 | 實務衝擊

**受影響產業**:
- 軟體開發產業（所有規模的軟體生產者）
- 政府機關（聯邦軟體採購方）
- 關鍵基礎設施（使用軟體系統的各部門）

**成本影響**:
- 軟體生產者需投資改善開發流程、導入安全實踐
- 採購方需建立供應商評估機制
- 培訓成本（開發團隊需學習新的安全實踐）

**技術挑戰**:
- 將高層次框架整合至現有 SDLC 模型
- 建立跨組織的共同安全詞彙
- 平衡開發速度與安全要求（特別是敏捷開發環境）

**預期時程**:
- 公開意見徵詢期：至 2026-01-30
- 最終版發布時程：待意見彙整後公告

---

# L4 | 衝突與不確定性

**模糊條文**:
框架提供「高層次實踐」，但具體實施細節需依組織特性調整，可能產生不同詮釋

**多方矛盾**:
- 開發速度 vs. 安全要求（DevSecOps 環境中的平衡點）
- 通用框架 vs. 產業特定需求（不同領域的軟體安全要求差異）

**待釐清問題**:
- 如何量化「減少漏洞數量」的成效
- 中小型軟體開發商的資源限制下如何落實完整框架
- 與其他軟體安全標準（如 ISO/IEC 27034）的協調機制
