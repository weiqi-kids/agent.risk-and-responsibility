- 風險已從個別工作負載轉移至控制平面（API、身份、政策、自動化）

# L4：落地執行指引

**雲端架構設計的根本轉變**：
- 從實體伺服器集合演變為由政策驅動的架構
- 存取權限在工作負載運行前即已定義
- 整個環境可透過單一 API 呼叫建立或銷毀

**三個平面的區別**：

| 平面 | 功能 | 重要性 |
|------|------|--------|
| 管理平面 (Management) | 帳戶層級治理 | - |
| **控制平面 (Control)** | **API、身份、政策、自動化** | **主要安全邊界** |
| 資料平面 (Data) | 實際工作負載運行 | - |

**風險轉移**：
- 攻擊者不再針對個別工作負載
- 目標轉為**呼叫特權 API、假冒特權角色、操控 IaC 管道**
- 控制平面遭入侵可導致「大規模、低噪音攻擊」

**Zero Trust 四大設計原則**：

1. **設計時期安全保證**
   - 安全性應在**設計階段宣告**，而非在運行時推斷
   - 角色、政策與審批邊界必須在任何 API 互動前預先定義

2. **工作負載間零信任**
   - 自動化身份應採用**最小權限**
   - 限定特定操作、環境與時間範圍
   - 不可透過帳戶成員資格或網路共置繼承存取權

3. **網路隔離是執行機制，非保證手段**
   - 網路控制：限制**可能的路徑**
   - 控制平面：定義**實際允許的行為**
   - 私有端點、分段等技術協助控制失效範圍，但不授予權限

4. **透過遙測持續驗證**
   - 控制平面活動往往與正常操作無異
   - 須透過以下信號持續確認：
     - 稽核日誌
     - 設定漂移偵測 (Configuration Drift Detection)
     - 政策合規性檢查
   - 「架構與政策的偏離，往往比傳統入侵警報更能指示風險」

**相關框架支持**：

| 框架 | 貢獻 |
|------|------|
| CSA CCM v4.x | 身份、治理、日誌、IaC 為基礎控制 |
| SCCF | 控制意圖、保證、漂移偵測 |
| SDP | 將存取與網路存在分離，聚焦身份 |

**核心結論**：
Zero Trust 在雲端的實踐，意味著將控制平面視為**主要安全邊界**——所有互動必須經過身份驗證、授權、政策約束，並**持續驗證**。安全保證來自系統設計，而非網路位置。

**實作建議**：
1. 在設計階段宣告安全性（角色、政策、審批邊界）
2. 為自動化身份實施最小權限與範圍限制
3. 以控制平面定義允許行為，網路控制僅限縮路徑
4. 建立稽核日誌、設定漂移偵測、政策合規檢查
5. 監控架構與政策的偏離作為風險指標
